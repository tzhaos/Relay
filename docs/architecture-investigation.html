Relay 应该是任务优先的 Rust 原生 AI 开发工作台

Orca 证明了多 Agent + 多 worktree + 原生 CLI 的产品方向；Zed 证明了 Rust + GPUI + Entity/Task 模型可以支撑高性能编辑器级体验。 Relay 的机会点，是把两者合并成一个为并行 Agent 开发原生设计的系统。

Executive Summary

一句话结论

Relay 不应该先做“编辑器”，而应该先做一个 Task-native runtime： 每个任务拥有 worktree、terminal session、agent status、diff review、preview context 和持久化历史。

建议的第一原则

Adopt

UI 风格遵循 Zed 的原生、克制、高密度设计；布局和功能对标 Orca 的三栏工作流。 底层不复制 Electron + Zustand 的状态拼装方式。 Rust 层应该有明确的一等实体：Task、Workspace、Worktree、 AgentSession、TerminalSession、DiffReview。

最小可行产品

Narrow

第一阶段只做五件事：打开 repo、创建 task worktree、在 worktree 中启动原生 CLI Agent、实时显示 changed files/diff、在任务列表展示 working/waiting/done。 不要在第一阶段做完整 LSP、插件市场、浏览器自动化、PR 面板和移动端。

Orca 对 Relay 的可借鉴清单
  • 保持 CLI 原生：不要吞掉 slash command、快捷键、TUI 输出和 shell 行为。
  • Worktree 创建、删除、归档、恢复必须有强安全校验，尤其要保护未提交改动和本地分支。
  • Agent 状态要走显式 adapter/hook/OSC 协议，终端标题只能作为弱 fallback。
  • Diff 评论、browser selection、PR review 都应挂在 task/worktree，而不是独立漂浮。
  • 远程能力需要从第一天设计抽象，但可以在 MVP 后实现。

Architecture Principles

Relay 的架构原则

1. Task 是一等实体

Task 不是 UI 行，也不是终端 tab。Task 是一个可恢复、可审查、可提交、可归档的工作单元。

2. Terminal 保持原生

CLI agent 的交互面由 PTY 真实承载。Adapter 只能监听、注入、标注，不重写 CLI 协议。

3. Worktree 隔离任务

每个 task 默认创建独立 Git worktree。所有 diff、preview、agent session 都以 task/worktree 为边界。

4. UI 读 projection，不拼底层状态

Rust core 产出 TaskProjection。GPUI 只负责展示和命令触发，避免 React/Zustand 式跨 store 合成。

5. Local/Remote 从接口层预留

MVP 只做本地，但 WorktreeProvider、PtyProvider、FsProvider 先设计成本地/SSH 可替换。

6. 安全边界前置

diff 评论、browser grab、agent prompt 注入、worktree 删除都要有预算、校验、审计日志。

UI Product Direction

Zed 风格，Orca 工作流

Relay 本质上是一个 Rust 高性能版 Orca：产品工作流、信息架构和功能优先级对标 Orca； 但视觉语言、交互质感和性能预期应对标 Zed，因为 GPUI 已经证明这种原生界面风格可以被 Rust 实现出来。

定位公式

Product

Zed-like surface, Orca-like workflow, Relay-native runtime. 也就是说，看起来和操作质感接近 Zed，功能布局和任务闭环接近 Orca，底层运行时则围绕 Relay 自己的一等 Task 模型重新设计。

维度 对标对象 Relay 决策
视觉风格 Zed 克制、原生、高密度、低 chrome；避免 Electron SaaS 感、重卡片、重阴影和花哨渐变。
交互质感 Zed / GPUI 键盘优先、快速响应、稳定焦点、轻量命令面板、面板切换无卡顿。
主布局 Orca 左侧 task/worktree/agent 状态，中间 terminal + preview，右侧 file tree / editor / diff / review。
功能闭环 Orca 多 Agent、多 worktree、原生 CLI、diff 评论、browser context、GitHub/Linear/PR 工作流。
底层架构 Zed 的 Rust/GPUI 方法论 Entity + Task + provider trait；UI 订阅 projection，core 负责状态机和生命周期。
产品差异 Relay 自己 Task 是一等实体，Agent session、worktree、terminal、diff、preview 全部归属 Task。
设计边界： 不要做 Zed 的完整替代品，也不要做 Orca 的像素级复刻。 Relay 应该像 Zed 一样快、稳、原生；像 Orca 一样懂 Agent 并行开发；但在任务模型和运行时上形成自己的架构。
Rust Ecosystem Strategy

充分使用小而精的 Rust 库

Relay 的核心竞争力不是重复实现成熟基础设施，而是把优秀 Rust crate 组合成稳定的 Agent-native 工作流。 除非某个能力直接决定产品差异化，否则优先采用已有库，并把精力放在任务编排、状态模型、UI 体验和安全边界上。

选型原则

Adopt
  • 优先选择单一职责、维护活跃、API 清晰、依赖可控的 crate。
  • 核心路径上避免庞大框架式依赖，除非它解决的是跨平台硬问题。
  • 对 Git、PTY、diff、watcher、SQLite、搜索、文本结构这类基础能力，默认不从零实现。
  • 把第三方库包在 Relay 自己的 provider/trait 后面，后续可替换，不让库 API 泄漏到业务模型。
  • 对安全敏感路径增加输入预算、路径校验、回滚和审计日志，而不是盲目信任库的默认行为。
能力域 优先借力方向 Relay 边界
PTY / Terminal portable-pty、alacritty_terminal、vte 一类成熟终端基础库。 封装为 PtyProvider 与 TerminalSession，UI 不直接依赖底层 PTY API。
Git MVP 优先调用 git CLI；结构化解析可逐步引入 gix 或轻量 parser。 封装为 GitProvider，所有 destructive 操作经过 safety gate。
文件监听 notify、ignore、walkdir 这类文件系统生态。 封装为 FsWatcher，输出去抖后的 worktree change event。
搜索 ignore + regex，或直接集成 ripgrep 思路；模糊搜索可参考 nucleo。 封装为 SearchService，先服务 task files、changed files、quick open。
Diff imara-diff、similar 等成熟 diff 算法库。 封装为 DiffEngine，业务层只消费 file/hunk/line projection。
文本结构 ropey、Zed 自有 rope 思路、tree-sitter 生态。 编辑器能力后置；MVP 只需要稳定读取、预览和 diff。
持久化 rusqlite、sqlx、redb 中择一，优先简单可靠。 封装为 repository 层，TaskEvent append-only，projection 可重建。
异步与通道 tokio 或 GPUI 集成 executor；通道使用 async-channel / flume。 后台任务必须可取消、可追踪、可回收，避免悬挂进程和泄漏订阅。
配置与序列化 serde、toml、schemars、figment 等。 配置 schema 版本化，用户配置和项目配置分层合并。
诊断与性能 tracing、tracing-subscriber、minidump、轻量 profiler。 所有 task/worktree/pty 生命周期都产生日志事件，便于复盘 Agent 运行。
设计约束： “用库”不等于把第三方抽象扩散到整个项目。Relay 应该在 core 里定义自己的 trait 和事件模型， 第三方 crate 只存在于 provider 实现层。这样既能快速启动，也能在性能、许可或跨平台问题出现时替换实现。

为什么不从完整编辑器开始

Decision

Zed 已经证明完整编辑器架构会迅速扩展到数百个 crate。Relay 的差异化不是再造 LSP/editor， 而是让多 Agent 并行开发变成稳定、可控、可审查的工作流。第一版可以嵌入轻量编辑/预览能力， 后续再逐步补 editor、LSP、symbol、semantic index。

第一批关键 trait 草案 
trait WorktreeProvider {
    fn create_worktree(&self, request: CreateWorktreeRequest) -> Task<Result<WorktreeSnapshot>>;
    fn remove_worktree(&self, request: RemoveWorktreeRequest) -> Task<Result<RemovalReport>>;
    fn changed_files(&self, worktree: WorktreeId) -> Task<Result<Vec<ChangedFile>>>;
}

trait PtyProvider {
    fn spawn(&self, request: PtySpawnRequest) -> Task<Result<TerminalSessionId>>;
    fn write(&self, session: TerminalSessionId, bytes: Vec<u8>) -> Result<()>;
    fn resize(&self, session: TerminalSessionId, cols: u16, rows: u16) -> Result<()>;
    fn kill(&self, session: TerminalSessionId) -> Task<Result<ExitStatus>>;
}

trait AgentAdapter {
    fn detect(&self, env: &RuntimeEnvironment) -> Task<Result<AgentAvailability>>;
    fn launch_plan(&self, request: AgentLaunchRequest) -> Result<AgentLaunchPlan>;
    fn parse_status_event(&self, event: TerminalEvent) -> Option<AgentStatusUpdate>;
}
实现纪律： core crate 不依赖 GPUI、不依赖 SQLite、不依赖具体 PTY/Git 库。 UI crate 不直接执行 git/pty/browser 操作。所有副作用通过 application service 调用 provider。
MVP Plan

四阶段路线

Phase 0：技术验证

GPUI window、三栏布局、SQLite 连接、git CLI probe、PTY spawn。目标是验证 Windows/macOS/Linux 基础可行性。

Phase 1：Task + Worktree + Terminal

打开 repo，创建 task worktree，启动 Claude/Codex CLI，显示任务状态和 changed files。

Phase 2：Diff Review 闭环

文件 diff、行级评论、发送评论到当前 Agent、标记 sent、生成 commit message。

Phase 3：Preview 和远程抽象

localhost preview、browser selection、安全 payload、SSH provider、远程 PTY/FS/Git。

// UI 层只调用 ApplicationService。
pub trait ApplicationService {
    fn open_project(&self, request: OpenProjectRequest) -> Task<Result<ProjectId>>;
    fn create_task(&self, request: CreateTaskRequest) -> Task<Result<TaskId>>;
    fn launch_agent(&self, task_id: TaskId, agent: AgentKind) -> Task<Result<AgentSessionId>>;
    fn send_to_agent(&self, task_id: TaskId, message: AgentMessage) -> Task<Result<()>>;
    fn archive_task(&self, task_id: TaskId) -> Task<Result<()>>;
}

许可风险

Important

Orca README 标注 MIT License；Zed README 标注主体 GPL-3.0-or-later，部分 Apache-2.0。 如果 Relay 不准备采用 GPL 兼容许可证，调研 Zed 时只能借鉴架构思想、模块边界和公开 API 使用方式， 不应复制 Zed 的 GPL 源码实现。

GPUI 版本风险

Watch

GPUI README 明确说明仍在 active development 且 pre-1.0，可能频繁 breaking change。 Relay 需要把 UI 层和 domain core 分离，避免 GPUI API 变化影响任务模型和运行时。

终端技术风险

Watch

真正难点不是 spawn 一个 shell，而是长期稳定处理 TUI、中文渲染、scrollback、resize、Windows ConPTY、进程树清理、恢复、远程复连。 建议第一版先选择成熟 PTY + 终端解析方案，再逐步优化渲染。
